# Metal/GPU Progress Summary

最終更新: 2026-04-23

## 目的

`Veloura Lucent` の補正とマスタリングについて、Metal/GPU 化の進捗を整理するメモです。

- 何を実装したか
- 何を保留にしたか
- 何が未実施か
- どの処理が Metal 向きか
- なぜそう判断したか

を、今のコードと実測ベンチに基づいてまとめています。

---

## 全体の結論

- 補正側は、**解析の一部を Metal 化して実運用に入っている**。
- マスタリング側は、**Metal 試作までは進んだが、本体接続はまだしていない**。
- 理由は、マスタリングの `spectralSummary` で
  - 速い版は解析値が少しズレる
  - CPU 一致版は速くならない
  という結果になったため。

---

## 1. 補正側の進捗

### 現状まとめ

- 状態: **一部完了**
- 実運用: **あり**
- UI 露出: **あり**
- 自動切替: **あり**

### 実装済み

| 処理 | 状態 | 向き | 理由 |
|---|---|---|---|
| 解析モード切替 (`auto / cpu / experimentalMetal`) | 完了 | 高 | Metal を使える Mac だけへ自動切替できるため |
| `computeMagnitudes` | 完了 | 高 | frame/bin 単位の並列計算だから |
| `computeTemporalMedian17` | 完了 | 高 | 各値の近傍中央値計算を GPU スレッドへ分散しやすいため |
| `separatedMeanSpectra` の一部 | 完了 | 高 | STFT 後の magnitude 系処理は並列化しやすいため |
| 補正ログの Metal 解析モード表示 | 完了 | 中 | 実際にどちらが使われたか確認しやすいため |

### 実測結果

実音源 `violin #002 睡眠.wav` での実測です。

```text
cpu.total: 370.945785s
experimentalMetal.total: 170.705045s
speedup.total: 2.173x

analyze: cpu=278.759780s
analyze: experimentalMetal=80.801004s
speedup.analyze: 3.450x

outputsMatch: true
```

### 保留

| 処理 | 状態 | 向き | 理由 |
|---|---|---|---|
| 補正処理本体の denoise / harmonicUpscale / loudnessFinalize の Metal 化 | 保留 | 中 | 解析ほど素直に並列化できず、音の変化リスクも上がるため |
| STFT 自体の Metal 化 | 保留 | 低〜中 | 既存の `vDSP` 置き換えは実装リスクが高いため |

### 未実施

| 処理 | 状態 | 向き | 理由 |
|---|---|---|---|
| 補正の true peak Metal 化 | 未実施 | 中 | reduce 系だが、今は補正側の主要ボトルネックではないため |
| 補正の stereo width Metal 化 | 未実施 | 中 | 補正本体の主経路ではなく、優先度が低いため |

## 2. マスタリング側の進捗

### 現状まとめ

- 状態: **試作まで**
- 実運用: **なし**
- 本体接続: **なし**
- 理由: **精度優先で採用条件を満たしていないため**

### CPU 側で完了した軽量化

これは Metal ではありませんが、現実に効いた改善です。

| 処理 | 状態 | 内容 |
|---|---|---|
| `loudness` | 完了 | 中間配列を減らして高速化 |
| `truePeak` | 完了 | 補間式を変えずにループと参照を軽量化 |
| `stereoWidth` | 完了 | 和差配列を作らず、左右二乗和と内積で計算 |

実音源ベンチの最近値:

```text
loudness: 1.161042s
truePeak: 1.910051s
stereoWidth: 0.004138s
```

### Metal 試作済み

| 処理 | 状態 | 向き | 理由 |
|---|---|---|---|
| `spectralSummary` の帯域集計 | 試作完了 / 本体未接続 | 高 | STFT 後の帯域合計は frame/bin 並列に向くため |
| `harshness` 集計 | 試作完了 / 本体未接続 | 高 | 周波数帯の magnitude 合計なので並列化しやすいため |

### 試作結果

#### 速い試作版

- 特徴: かなり速い
- 問題: CPU と解析値が少しズレる

確認できた差分:

```text
diff.lowBand: 0.000437 dB
diff.midBand: 0.018861 dB
diff.highBand: 0.071198 dB
diff.harshness: 0.000000
```

この差分は小さく見えますが、ユーザー要件が「音を変えてまでやらない」なので、本体接続は見送りました。

#### CPU 一致を詰めた版

- 特徴: 差分はほぼゼロ
- 問題: 速くない

直近の実測:

```text
cpu.spectralSummary: 4.317682s
experimentalMetal.spectralSummary: 4.797733s
speedup: 0.900x

diff.lowBand: 0.000000 dB
diff.midBand: 0.000001 dB
diff.highBand: 0.000000 dB
diff.harshness: 0.000000
```

つまり、

- **速い版**: 速いが少しズレる
- **一致版**: 一致するが速くない

という状態です。

### 保留

| 処理 | 状態 | 向き | 理由 |
|---|---|---|---|
| `spectralSummary` Metal 版の本体接続 | 保留 | 高 | 速い版は差分あり、一致版は遅いため |
| `harshness` を含む Metal マスタリング解析モード | 保留 | 高 | 実験モードとしては可能だが、採用条件を満たしていないため |

### 未実施

| 処理 | 状態 | 向き | 理由 |
|---|---|---|---|
| `truePeak` の Metal 化 | 未実施 | 中 | 並列 reduce は可能だが、実装が少し重く、CPU 軽量化が先に効いたため |
| `loudness` の Metal 化 | 未実施 | 中 | prefix / 窓処理が必要で、GPU に寄せても構造が複雑になるため |
| `stereoWidth` の Metal 化 | 未実施 | 中 | CPU 側の Accelerate 化で十分軽くなったため |
| STFT 自体の Metal 化 | 未実施 | 低〜中 | `vDSP` 置き換えのリスクが高いため |

## 3. 向き不向きの整理

### Metal 向き

| 処理 | 向き | 理由 |
|---|---|---|
| STFT 後の帯域集計 | 高 | frame/bin ごとの並列計算に向くため |
| harshness 集計 | 高 | 周波数帯の合計なので並列化しやすいため |
| 補正解析の magnitude / temporal median | 高 | 同じ形の計算を大量に回すため |

### 条件付きで向く

| 処理 | 向き | 理由 |
|---|---|---|
| true peak | 中 | 並列 reduce はできるが、補間ピーク値の一致確認が重い |
| loudness | 中 | prefix / 窓 / gate があり、単純な GPU 化では済まない |
| stereo width | 中 | reduce 系なので可能だが、CPU Accelerate でも十分軽い |

### あまり向かない

| 処理 | 向き | 理由 |
|---|---|---|
| STFT 自体 | 低〜中 | 既存の `vDSP` が強く、置き換えリスクが高い |

## 4. 今の判断

### 補正

- **Metal は採用済み**
- 理由:
  - 実測で十分速い
  - 出力 A/B 一致確認済み
  - UI とログでも切替状況を確認できる

### マスタリング

- **Metal は未採用**
- 理由:
  - 速い版は差分が出る
  - 一致版は速くない
  - 音質・解析値優先の条件にまだ合わない

## 5. 次にやるなら

優先度順に並べると、今はこうです。

1. マスタリング `truePeak` の追加軽量化  
   CPU でまだ約 1.9 秒かかるため

2. `spectralSummary` Metal 版は保留のまま設計メモ化  
   速い版と一致版の差を今後比較しやすくするため

3. `loudness` の追加 CPU 軽量化  
   今も約 1.16 秒あるため

4. STFT の Metal 化は後回し  
   置き換えリスクが高いため

---

## 6. 一言まとめ

- 補正の Metal 化: **進んでいる。実運用あり**
- マスタリングの Metal 化: **試作は進んだが、本採用はまだ**
- 今の基準: **速度より、まず CPU 一致と音の安全性**
