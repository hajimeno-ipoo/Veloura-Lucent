# Veloura Lucent の概要

## これは何をするアプリか
- 音声ファイルを読み込みます。
- まず、耳につくザラつきやシュワシュワ感を減らします。
- ノイズ除去は `弱い / 標準 / 強い` から選べます。
- そのあと、高い音の不足を少し補います。
- 高い音の補い方では、倍音寄りの成分と立ち上がりを見て、16kHz 以上の伸びを作ります。
- 高い音をどれだけ足すかは、軽い `foldover補完量推定` が自動で決めます。
- `foldover補完量推定` は、高域不足、16kHz以上の残り具合、18kHz以上の荒れも見て、足しすぎを避けます。
- ノイズ除去では、低中域の音楽成分があり、静かな区間より高域が自然に伸びている時は、8kHz〜20kHzの煌びやかさ、空気感、超高域をできるだけ残します。
- ノイズ除去では、静かな区間に出るヒスや、音楽成分と一緒に動いていない高域ノイズは下げます。高域保護は基本のノイズゲートにだけ使い、チリつき確認や最終mask全体を無条件に持ち上げません。
- ノイズ除去では、完全な静音部は今まで通り下げます。ただし、静音部より少し強く、演奏の余韻として残っているフレームでは、60Hz〜150Hzと150Hz〜250Hzを少し守り、低域と低中域が吸い込まれすぎないようにします。
- ノイズ除去では、10kHz〜16kHzの一瞬だけ出るチラつきも見て、AI生成音源のシュワシュワ感を抑えます。
- サ行とシマーの保護では、8kHz以上を常に下げず、短時間だけ大きく出るサ行、刺さり、チリつきだけを下げます。
- サ行保護後に5kHz〜9kHzの通常の明るさが下がりすぎ、サ行だけが相対的に目立つ時は、短時間ピーク以外の通常高域を戻し、まだ強い短時間ピークだけを軽く抑えます。
- そのあと、別機能のマスタリングで仕上がりを整えます。
- 補正後と最終版は、用途に合わせて `高品質保存`、`配信・納品用`、`CD用`、`試聴共有用` から選んで書き出せます。
- 内部のプレビュー用ファイルは、音の情報を保つため `32-bit float WAV / 48kHz` のままです。
- 48kHz以外の音声を読み込む時は、OS標準の音声変換で48kHzにしてから処理します。
- `配信・納品用` は `24-bit PCM WAV / 48kHz`、`CD用` は `16-bit PCM WAV / 44.1kHz + TPDFディザ`、`試聴共有用` は `AAC .m4a / 48kHz / 256kbps` です。
- 補正段では最終音量を作らず、ノイズ除去、低域整理、最低限の高域修復、ピーク保護だけを行います。
- 補正段では、ハム、低域ゴロゴロ、サ行、シマー、ヒスを順番に見て、強い設定ほど実測で下がるようにしています。
- 補正段の低域ゴロゴロ整理では、静かな区間の20Hz〜150Hzは今まで通り下げます。ただし、標準補正の演奏中と余韻では60Hz〜150Hzの削り量だけを少し弱め、低域の土台が細くなりすぎないようにします。20Hz〜60Hzはハムやゴロゴロ感が戻りやすいため大きく戻しません。強い補正では、ハムと低域ゴロゴロをしっかり下げる目的を優先します。
- 補正段の高域保持では、8kHz〜16kHz の実際の音量が原音より少し痩せた時に、ヒス、シマー、サ行、16kHz以上の超高域が戻りすぎない範囲で少し戻します。
- マスタリングでは、低域、中低域、プレゼンス帯域、エアー帯域、ハーシュネスをまとめて整えられます。
- 標準マスタリングでは、音量を前に出しすぎず、8kHz〜12kHz のきらめきをサ行の瞬間成分から分けて少し戻します。
- 標準マスタリングでは、低域と部屋鳴りが戻りすぎないよう、低域と中低域の持ち上げを控えめにしています。
- マスタリングのノイズ戻りガードでは、静かなヒスと短時間の高域チラつきを分けて見ます。持続する8kHz〜16kHzの音楽的な高域は、まとめて暗くしないように守ります。
- マスタリングでは、高域戻りガードを通常時は使わず、刺さりと高域戻りが同時に強い時だけ使います。
- マスタリングでは、ノイズを減らし続けるよりも、原音の透明感と空気感を残すことを優先します。
- マスタリングでは、最後のヒス・サ行・高域チラつき確認も緊急用に寄せ、少しの戻りで高域全体を削りすぎないようにしています。
- マスタリングでは、`targetLoudness` を必ず合わせる数値ではなく、仕上がりの音量目安として使います。安全に届く時は目標音量へ近づけ、音量変更は `自然`、`聴きやすく整える`、`押し出し強め`、`安全AI配信`、`YouTube / Spotify向け`、`リリース音圧重視` ごとの上限内に収めます。
- マスタリングでは、ノイズ戻りガードや高域保持のあとに最終音量を再確認します。目標音量との差だけで判断せず、True Peakの余裕、プリセットごとの音量変更上限、プリセットごとの目安超過幅、ノイズ戻りを守れる時は、最大2.0dBまで最終音量を戻します。目安超過幅は、自然と安全AI配信が0.75dB、聴きやすく整えるが1.0dB、YouTube / Spotify向けが1.25dB、押し出し強めが1.5dB、リリース音圧重視が2.0dBです。音量復帰後の全体確認で、補正後よりヒスや高域チラつきが種類ごとの上限を超えて戻った場合は、緊急上限ガードで抑え直します。ヒスは復帰前も確認し、復帰前が上限内なら音量復帰前へ戻します。補正後で高域が大きく失われ、原音参照で高域を戻す素材でも、補正後からのヒスや高域チラつきの戻り上限は守ります。復帰前も上限を超える場合は緊急上限ガードを続けます。
- マスタリングでは、8kHz〜12kHz の煌びやかさと 12kHz〜16kHz の空気感を、軽い帯域測定で確認し、最終版だけ暗くなりすぎないようにしています。
- マスタリングでは、補正後にサ行が強く残った素材だけ、最後に 5kHz〜9kHz の短時間ピークを軽く抑え、原音からの増えすぎを防ぎます。
- マスタリングログでは、ラウドネス、高域戻りガード、ノイズ戻りガードを分けて表示し、どこに時間がかかったか見やすくしています。
- マスタリングでは、低域は広げすぎず、中高域だけ広げるようにして、芯が痩せにくい形にしています。
- マスタリングでは、ラウドネス調整後に演奏中と余韻の20Hz〜1kHzだけを確認し、低域と低中域が下がりすぎた時だけ上限付きで少し戻します。60Hz〜150Hzは低域の土台として分けて見ます。150Hz〜500Hzは演奏中フレームだけ最大0.5dB程度まで軽く戻し、胴鳴りと厚みが薄くなりすぎないようにします。この150Hz〜500Hzの確認は補正後の音を基準にします。原音がある時も、150Hz〜500Hz以外の低域保護で原音を参考にし、完全に戻すのではなく削れすぎた分だけ自然に軽く戻します。さらに、最終ノイズ確認後にも、補正後より150Hz〜500Hzが下がっている時だけ、演奏中の150Hz〜500Hzを最大0.5dB戻します。最後にプリセットごとの音量上限を確認します。静かな区間のノイズ除去は弱めません。
- マルチバンド圧縮は、低域・中域・高域を整理しつつ、超高域は潰しすぎないように分けています。
- 詳細設定では `ダイナミクス保持` と `仕上げの強さ` で、平坦さと押し出し感をまとめて調整できます。
- 入力・補正後・最終版の差は、中央の作業画面で続けて確認できます。
- 中央の作業画面では、3つの音源を横長の波形で縦に並べます。波形をクリックまたはドラッグすると、3つの音源の再生位置を同じ時間へ動かせます。
- 中央の作業画面では、再生中スペクトルを1つのグラフに表示します。入力、補正後、最終版を再生またはA/B切り替えした時に、いま鳴っている音の周波数ごとの強さを見ます。
- 再生中スペクトルの近くには `ベクトルスコープ` を表示します。いま再生している1音源だけを対象に、`Polar Sample`、`Polar Level`、`Lissajous` を標準のセグメント切り替えで選び、左右チャンネルの広がり、向き、位相ずれを確認します。停止中は軌跡を残さず、枠と説明だけを表示します。
- `Polar Sample` では半円型の表示に、サンプルごとの点、45度安全ライン、3秒で薄く消える履歴を表示します。クリップした点は赤く表示します。`Polar Level` では平均化した線を表示し、線の角度でステレオ位置、線の長さで振幅を見ます。線の履歴は3秒で薄くなりながら中心へ縮みます。右側には相関メーター、下側にはL/Rバランスメーターを表示します。
- ベクトルスコープの右側には `ラウドネスメーター` を表示します。再生中の音から Momentary LUFS、Short-Term LUFS、Integrated LUFS、True Peak dBTP を測り、目標LUFSとTrue Peak上限も同じ場所で確認します。単なる数値だけではなく、縦型メーターで現在値、目標線、上限線を見ます。
- 中央の作業画面では、スペクトログラムを横長の3段で表示します。時間ごとの高域、低域、ノイズの出方を同じ画面で見ます。色は3音源で共通の `0〜-100表示dB` を使い、右側の凡例で強さを確認できます。
- 品質チェックは右設定パネルにまとめています。音量低下、ピーク超過、高域の増えすぎ、ステレオ幅の急変、音の起伏の変化を知らせます。
- 右設定パネルでは、確認する音源を `入力 / 補正後 / 最終版` から選び、ラウドネス、True Peak、ダイナミクス、ステレオ幅を見ます。
- ノイズチェックは1つのカードにまとめ、入力、補正後、最終版のノイズ量を横棒で見せます。各行には「原音より低い」「少し戻りあり」などの判定も出します。
- ノイズチェックの横棒は実測値そのものではなく、見た目で差が大げさになりすぎない表示用スケールで出します。右側の数値と差分は実測値のままです。
- ヒスと高域のチラつきの横棒は、警告しきい値まで同じ目盛りに入れています。悪い値が全部同じ最大表示に見えないようにするためです。
- 未測定のノイズ行は、同じ長さの薄いグレーの棒で表示し、まだ測っていないことが分かるようにします。
- ノイズチェックのおすすめ設定は固定幅ではなく、補正で減った量、マスタリングで戻った量、残っているノイズ量から計算します。すでに上限や下限で動かせない項目は表示しません。
- 左サイドバーは `音源` と `工程` の2つに分けています。`音源` では入力、補正後、最終版それぞれのファイル名、保存場所、形式、サンプルレート、チャンネル、長さを同じ行の中で確認できます。`工程` では補正とマスタリングそれぞれの工程を確認できます。
- 左サイドバーの `工程` では、補正とマスタリングを別々に表示します。補正は16工程、マスタリングは19工程を上から順番に出し、完了、実行中、省略、失敗、待機をアイコンと文字で見分けられるようにしています。
- ファイル情報は実際の音声ファイルから取得します。WAVは取得できたビット深度を表示し、MP3やM4Aは存在しないビット深度を作らず、MP3またはAACと表示します。
- 固定下部では、左に直近の処理内容を最大3件、右に入力解析、補正処理、マスタリング、書き出しの全体工程を表示します。
- 直近の処理内容には、時刻、処理名、対象ファイル、取得できた解析値、処理中の進捗率を、内容に応じて表示します。完全な処理ログは別に残します。
- 全体工程は、補正またはマスタリングの実際の進捗率と連動します。待機中、処理中、完了の位置を4段階で確認できます。
- 詳細ログボタンを押すと、補正ログとマスタリングログの全文を同じ幅の2列で確認できます。長い行は折り返し、縦にスクロールできます。
- 解析表示で測ったLUFSは、プレビューの音量合わせ再生にも使います。プレビュー側で同じラウドネスをもう一度重く測らないためです。
- 解析表示でプレビュー帯域とスペクトログラムを同時に作る時は、STFT全体を大きな配列として持たず、1フレームずつ帯域値と表示セルを集計します。同じ音声から同じ周波数解析を2回作らず、表示用メモリも減らすためです。
- 解析表示は、比較、スペクトログラム、プレビュー、ノイズ確認、補正解析、マスタリング解析を別々の完了状態で扱います。重いノイズ確認を待たずに、先に終わった表示から画面へ出すためです。
- 解析表示の時間計測と、重い表示解析を画面の外で動かす補助処理は `DisplayAnalysisSupport.swift` に分けています。表示解析の計測と別タスク実行を、画面本体から分けて扱うためです。
- 解析表示の完了状態、実行中状態、失敗状態は `DisplayAnalysisStateStore.swift` に分けています。入力、補正後、最終版ごとの表示解析状態をまとめて扱うためです。
- 補正とマスタリングの進捗状態は `ProcessingProgressStateStore.swift` に分けています。実行中、完了、スキップ、失敗の状態更新を同じ仕組みで扱うためです。
- 補正設定、マスタリング設定、解析モード、処理時に固定した設定は `ProcessingSettingsState.swift` に分けています。設定変更と処理状態を分けて扱うためです。
- 新しい音源や新しい処理結果を扱う時は、古い表示解析タスクをキャンセルします。補正後のマスタリング解析とノイズ確認がそろうまでは、マスタリング開始ボタンを押せないようにしています。
- 試聴状態は、入力、補正後、最終版ごとに再生位置を持ちます。A/Bを切り替えても同じ時間から聞けるようにします。
- 補正ログとマスタリングログは1行ずつ配列で保持し、通常画面では最新80行だけを描画します。ログ全文は内部で残しつつ、長いログ全体を毎回描き直さないためです。
- 比較メトリクスでは、周波数帯の範囲を先に決めておき、解析中に余計な配列を作りすぎないようにしています。
- 再生中スペクトルでは、`AVAudioEngine` の再生経路から音声バッファを受け取り、`Accelerate` で周波数ごとの強さを計算します。曲全体の平均値ではなく、いま再生している部分の音を表示するためです。
- 再生中スペクトルの画面表示は `Swift Charts` ではなく `Canvas` で描きます。再生中に値が更新されても重くなりにくくするためです。Y軸は固定したdBFS目盛りにし、測定点を滑らかな曲線でつないで周波数ごとの変化を見やすくします。
- ベクトルスコープは、再生中スペクトルと同じ `AVAudioEngine` の音声受け取り口を使います。Lissajous は横方向を `(右 - 左) / 2`、縦方向を `(左 + 右) / 2` として計算し、菱形の枠で表示します。Polar Sample は横方向を `(右 - 左) / 2`、縦方向を `abs((左 + 右) / 2)` として半円型に表示し、45度安全ラインで位相ずれを確認します。Polar Level は短い区間の中央成分と左右差成分の平均から線を作り、線の角度でステレオ位置、線の長さで振幅を確認します。別の音声受け取り口を増やさないため、再生処理へ余計な負担を重ねません。
- ベクトルスコープの計算は音声再生を止めない別の処理列で行い、画面の更新だけをメイン処理へ渡します。履歴は薄く残し、停止、自然終了、ファイル変更では古い軌跡を消します。一時停止では、停止した瞬間の軌跡を保持します。
- ラウドネスメーターも同じ `AVAudioEngine` の音声受け取り口を使います。別の受け取り口を追加せず、計算は別の処理列、画面更新はメイン処理で行います。
- 比較メトリクスでは、マスタリング解析を丸ごと再実行せず、必要な数値だけを直接計算します。
- 比較メトリクスでは、音量窓の計算に累積エネルギーを使い、同じ波形を何度もなめないようにしています。
- LUFS、True Peak、LRA、短い時間ごとの音量は `LoudnessMeasurementService` でまとめて測ります。表、グラフ、品質チェックで別々の基準にならないようにするためです。
- Integrated LUFSは、公式K-weighting係数、400ms窓、100ms間隔、絶対ゲート、相対ゲートを使って測ります。外部のEBU R128系メーターとの差を小さくするためです。
- Short-Term LUFSは、3秒前後の窓を公式K-weightingのエネルギーから計算します。単なるRMS dBではなく、LUFSとして読める値にそろえています。
- True Peakは、サンプルだけの最大値ではなく、4倍オーバーサンプリングのFIR補間でサンプル間ピークも確認します。画面では dBTP として表示します。
- 比較表のLRAは、48kHzのモノラルとステレオだけを対象に、3秒窓、0.1秒間隔、末尾1.5秒の無音、絶対ゲート、相対ゲートを使って測ります。正式な条件で測れない時は `--` と表示します。再生音声と保存WAVへ無音は追加しません。
- LUFSだけが必要な処理では、True Peak、LRA、短い時間ごとの音量を同時に計算せず、LUFSに必要な計算だけを実行します。
- FFT処理では、フレームごとの作業配列を使い回して、短い配列を何度も作らないようにしています。
- 逆FFT処理でも、フレームごとの作業配列を使い回して、波形へ戻す処理を軽くしています。
- foldover用の逆FFTでは、全体を毎回ゼロにせず、使った周波数ビンだけを片付けます。
- スペクトログラムは、フレームごとの小さな配列をたくさん持たず、1本の連続した配列で持ちます。
- 解析と表示では、スペクトログラム全体の大きな二重配列を作らず、必要なフレームやビンだけを読みます。
- 解析で使う小さい中央値フィルタは、結果を変えずに短い配列生成を減らしています。
- foldover系の高域補完では、全フレーム分の空スペクトログラムを作らず、1フレームずつ逆FFTへ流します。
- 比較メトリクスの非同期処理では、波形系、チャンネル系、FFT系を大きな単位で並列実行します。
- ノイズ除去では、ビンごとの固定係数を先に計算し、mask式の結果を変えずに重複計算を減らしています。
- ノイズ除去のmask適用では、フレーム配列を直接参照して、同じ式のまま添字アクセスを減らしています。
- ノイズ除去の高域保護量は、フレームとビンごとに先に計算し、mask適用中は同じ値を配列から読みます。高域保護の式と出力は変えていません。
- ノイズ除去の低域保護は、強い演奏中フレームと余韻フレームを分けます。余韻フレームでは60Hz〜150Hzと150Hz〜250Hzだけ最低mask量を上げ、20Hz〜60Hzや250Hz以上は大きく戻さないようにしています。
- ハム測定では、50/60Hzと倍音の周辺比の判定式を変えず、同じフレーム内の同じ周波数だけを1回測って使い回します。STFTのreal/imag全体は保持せず、1フレームずつ周辺比を集計します。
- ハム除去では、静かな区間は今まで通り削り、演奏中は削り量を弱めます。ただし、50/60Hz系の中心周波数が周辺より明確に大きい時は、演奏中でもハムとして削ります。
- 低域ゴロゴロ整理では、60Hz〜150Hzだけフレームの強さで削り量を変えます。静かな区間は通常どおり、余韻は中間、演奏中は控えめに削ります。20Hz〜60Hzと150Hz以上は従来どおり扱います。
- パーセンタイル計算では、返す値を変えず、全体を並べ替えずに必要な順位の値だけを選びます。
- ノイズ測定の帯域通過フィルタでは、highPassとlowPassの式、順番、4回反復を変えず、作業用配列と係数を使い回して同じ値を返します。
- ノイズ除去では、10kHz〜16kHzの前後フレームを比べ、急に出る高域の粒子感だけを少し安定させます。
- 診断用の出力先を渡した時は、ノイズ除去の8kHz〜12kHz、12kHz〜16kHz、16kHz〜20kHzについて、raw mask、granular mask、shimmer mask、最終maskが何dB下げたかをログと診断レポートに出します。
- 補正後高域保持は、全体音量との比率ではなく、8kHz〜12kHz と 12kHz〜16kHz の実レベル低下を見て、16kHz〜20kHz が戻りすぎない候補を選びます。
- 補正後高域保持では、まず補正候補そのものを安全確認します。問題がなければ、音を薄めた候補は作らず、同じ音の結果を保ったまま余計な計算を減らします。
- 補正後高域保持では、原音側の4つの高域測定を先にまとめて計算します。補正中に変わる音側は、その時点の音を今まで通り測ります。
- マスタリングのノイズ戻りガードは、ラウドネス補正後の高域ピークだけで判断せず、静かな区間のヒス床と短時間だけ不安定に出る高域を分けて判定します。
- 補正にも詳細設定があります。弱い、標準、強いの各プリセットが初期値を持ち、補正の強さ、原音保持、低域整理、中低域整理、高域補完、高域の自然さなどを調整できます。
- 補正ログには、ノイズ除去だけで10kHz〜16kHzのチラつき、12kHz以上、16kHz以上、18kHz以上が何dB変わったかを表示します。
- 画面の比較エリアには、ノイズチェックも表示します。ヒスは静かな区間の8kHz以上、サ行は5kHz〜9kHzの瞬間差、ハムは50/60Hzと倍音の周辺比のように、種類ごとに測り方を分けます。
- ノイズチェックの各行では、入力、補正後、最終版の実測値、補正での改善量、マスタリングでの戻り量、総合判定を同じ横棒グラフの中で確認できます。
- マスタリングでは、ハーシュネスが高く、かつ高域戻りも強い素材だけ、10kHz以上を弱く抑える高域戻りガードを入れています。
- マスタリング解析では、STFTの全フレーム結果を大きな配列として持たず、1フレームずつ読みながら帯域レベルと刺さり感を集計します。Metal対応Macでは、数百フレームごとにMetalへ渡して同じ結果を出します。使えない時だけCPUで同じ計算をします。
- マスタリング解析のラウドネス計算では、中間配列を減らして同じ結果のまま軽くしています。
- マスタリング解析のステレオ幅計算では、和差の配列を作らず、左右の二乗和と内積から同じ値を出しています。
- マスタリング解析では、STFTと帯域集計、ラウドネス、トゥルーピーク、ステレオ幅の工程別ベンチを取り、マスタリングログにも表示します。
- マスタリング解析では、実音源でも工程別ベンチを取り、どの工程が一番重いか確認できます。
- マスタリング解析の帯域集計では、Metalを本処理に使い、ログでは `STFT+帯域集計(Metal)` と `帯域集計(Metal)`、またはCPU版の名前で表示します。
- トゥルーピーク解析では、補間式を変えずに配列生成と内側ループの余分を減らしています。
- 補正処理では、左右チャンネルの重いSTFT処理を並列に進めて待ち時間を減らします。
- 補正処理では、読み込み、解析、低域ノイズ整理、ノイズ除去、サ行保護、再解析、高域修復、低中域残り、ピーク保護、書き出しの工程別ベンチを取れます。
- 入力ファイル選択時に作った補正用解析結果は、解析モードが同じなら補正開始時に使い回します。解析がまだ無い時や解析モードが変わった時は、従来通り補正内で解析します。
- 入力や補正後のノイズ測定値がすでにある時は、補正の低域整理・シマー制限、マスタリングのノイズ戻りガードで使い回します。
- 補正とマスタリングは、開始直後に素材を診断し、各工程を `実行 / 軽量 / スキップ` に分けます。
- スキップした工程は、ログと進行状況の両方に表示します。
- 診断用の出力先を渡した時だけ、補正とマスタリングの工程ごとの中間WAVを書き出せます。
- 中間WAVは、どの工程で高域、空気感、音量感が変わったかを、実音源で確認するために使います。
- 進行状況はログの文章ではなく、処理側から出す工程イベントで動きます。文章を少し変えても、進行表示が壊れにくい作りです。
- 補正ログとマスタリングログは、専用のログ状態ストアで全文と画面表示用の直近80行を管理します。
- 長い工程では、代表区間の確認回数などを進行状況にも出し、止まって見えないようにします。
- 補正完了とマスタリング完了では、設定の「完了通知」がONの時だけ、音付きローカル通知を1回だけ送ります。
- `ノイズ除去`、`高域修復`、`ピーク保護`、`ラウドネス` は安全上の重要工程なので常に実行します。
- 高域修復後のシマー保護は、修復でヒス、サ行、シマーが実際に増えた時だけ実行します。
- サ行/シマー保護は、5kHz〜9kHzのサ行、6kHz〜10kHzの刺さり、10kHz〜14kHzのチリつきを別々に測り、前後フレームより短時間だけ大きい部分を下げます。
- 8kHz〜12kHzの持続する煌びやかさと、12kHz〜16kHzの空気感は、サ行/シマー保護で常時削らない設計です。
- プレビュー用の一時ファイルは `VelouraLucentPreview` に作り、新しい入力選択時と画面終了時に掃除します。
- 画面表示用の解析は、既に入力・補正後・最終版の結果がそろっている時は再実行せず、キャッシュ済みの結果を使います。
- 補正の低域ノイズ整理とシマー制限は、測定回数をログに出します。シマー制限は、静かなヒス床が低くても、8kHz〜14kHzで短時間だけ飛び出るチラつきがあれば実行します。
- シマー制限は、長い音源で全体測定を何度も繰り返さないよう、代表区間で軽く測り、強い補正時だけ最後に全体確認を1回行います。16kHz以上の空気感は戻して守ります。
- マスタリングのノイズ戻りガードも、長い音源で全体測定を何度も繰り返さないよう、代表区間で軽く測り、必要な時だけ最後に全体確認を1回行います。
- マスタリングのノイズ戻り候補を確認する時は、変化しない基準音の8kHz〜20kHzを候補ごとに測り直さず、先に測った3帯域の値を使い回します。候補音側は、その時点の音を今まで通り測ります。
- マスタリングのノイズ戻り判定は、共通の軽量測定と高域保護を `MasteringNoiseReturnSupport.swift` に分けています。最後の緊急ノイズ上限確認は `MasteringFinalNoiseReturnCeiling.swift` に分けています。音の計算順、しきい値、ログ、診断WAV名は変えていません。
- マスタリングでは、ヒスを抑えすぎて5kHz以上の明るさが大きく落ちないよう、最後に高域の残り方も確認します。
- 補正とマスタリングの内部判定は `InternalAudioJudgementPolicy` に主要なしきい値をまとめ、意味が違う判定値は別の定義として扱います。
- ノイズ測定値が欠けた時は「問題なし」と見なさず、再測定できる工程では再測定し、ルート判定では処理をスキップしない方向で扱います。
- 補正処理のピーク保護では、音量を持ち上げず、クリップしそうな部分だけ安全側へ抑えます。
- 補正処理の高域補完では、12kHz〜16kHzの落ち込みだけでなく、16kHz以上がすでに残っているか、18kHz以上に荒れが多いかも見て、補完量を調整します。
- 帯域処理では、帯域エネルギーとgain計算の作業配列を使い回し、短い配列生成を減らしています。
- `STFT/ISTFT` の往復削減は音質リスクが高いため、先に `Docs/STFTISTFTRoundTripReductionDesign.md` で設計を固定しています。
- 解析のMetal/GPU化は、先に `Docs/AudioAnalysisMetalGPUDesign.md` で実験モード前提の設計を固定しています。
- `temporal median` のMetal化は、先に `Docs/TemporalMedianMetalDesign.md` で一致条件と採用条件を固定しています。
- 解析には `自動`、`安定CPU`、`実験Metal` の入口があり、UIから選べます。初期値は自動です。
- 補正用解析では、STFTのreal/imag全体を持たず、周波数ごとの大きさだけを保持して解析します。実験Metalでは `temporal median 17` をMetalで試作しています。マスタリング解析の帯域集計は、Metal対応Macでは本処理でMetalを使います。
- 用語は正式名称へ寄せて、`?` ボタンから読み方と意味を確認できます。
- ブラウザではなく、Mac の普通のアプリとして動きます。
- 画面は Apple の Liquid Glass UI 方針に合わせ、`macOS 26` 以上を前提にしています。
- 画面全体は、左の入力と処理状態、中央の横長波形・再生中スペクトル・ベクトルスコープ・スペクトログラム、右の確認値と詳細設定、下の直近ログと全体工程に分かれます。
- 上部ツールバーでは、音声選択、補正実行、マスタリング実行、書き出しを中央の処理操作としてまとめ、右設定の表示切り替えだけを右端に分けています。
- 詳細設定は右側の設定パネルにまとめています。横に広い旧設定画面をそのまま入れず、右側で読めるように1項目ずつ縦に並べています。
- 右設定パネルは必要な時だけ表示できます。閉じると中央作業画面が広がり、左側の入力と処理状態は表示されたまま残ります。
- 試聴比較は中央上部にまとめています。入力、補正後、最終版を聞き比べ、A/B比較、再生、一時停止、停止、音量、ラウドネス合わせを同じ場所で操作します。
- 左側のファイル名と保存場所は、長くてもサイドバーを押し広げないよう途中を省略して表示します。マウスを重ねると完全な保存場所を確認できます。
- 中央は `基本表示 / 詳細解析` で切り替えます。`基本表示` では、波形、再生中スペクトル、ベクトルスコープ、スペクトログラムを同じ縦の流れで表示します。
- `詳細解析` では、入力、補正後、最終版の主要数値、補正差分、マスタリング差分、ノイズ7種類、ステレオ相関、時間ごとのステレオ相関推移、短時間ラウドネス、ダイナミクス推移、平均スペクトル、周波数帯域詳細を確認できます。
- ステレオ相関は、上段で曲全体の値を見ます。下段では、曲のどの時間で左右の音が逆向きに近づいたかを見ます。0より下の時間帯は、モノラル再生で音が痩せる可能性があります。無音区間は相関値として計算せず、推移線もそこで区切ります。横軸は音源全体の長さで表示するため、末尾の無音区間も線がない時間として残ります。モノラル音源では推移グラフではなく説明文を出します。
- `仕上がりの方向` は独立カードに戻さず、ラウドネス、True Peak、ダイナミクスは主要数値比較へ、煌びやかさ、空気感、超高域、こもりは周波数帯域詳細へ統合します。良否や注意は右側の品質警告で確認します。
- 詳細解析の表は、右設定パネルや下部ログへ重複して出しません。右側は設定、選択中音源の数値、品質警告、完了レポートを担当し、下部は直近ログと全体工程を担当します。
- 左側は狭いサイドバーなので、3音源それぞれのファイル情報を音源行の中に入れます。補正とマスタリングの工程一覧、実行時間、進捗率は `工程` にまとめます。補正とマスタリングは1つの進捗にまとめず、別々に確認できるようにしています。
- 試聴の音量、ラウドネス合わせ、現在の比較対象は、横幅が足りない時に上下へ並びます。
- 補正中やマスタリング中に触ると危ない操作は、ツールバーや設定画面でも押せないようにしています。
- Liquid Glass の見た目は、標準のサイドバーとツールバーを優先して使います。右設定は、中央作業画面と同じ詳細側の中で開閉できるパネルとして表示します。独自の見た目は、再生操作や状態確認のように意味がある場所だけに使います。

## 技術アーキテクチャの概要
- 画面: `SwiftUI`、左サイドバー付きの `NavigationSplitView`、中央作業画面、開閉式の右設定パネル、`toolbar`
- 音声ファイルの読み書き: `AVFoundation`
- 周波数ごとの計算: `Accelerate`
- 実行入口: `script/build_and_run.sh`

図にすると、こうです。

```text
Macアプリ(SwiftUI)
  -> AudioFileInfo
    -> 入力、補正後、最終版の形式、サンプルレート、チャンネル、長さ
    -> 取得できる時だけWAVのビット深度
  -> RecentActivityEvent
    -> 選択、解析、補正、マスタリング、書き出しの直近処理
    -> 時刻、対象ファイル、解析値、進捗率
  -> NotificationService
    -> 完了時のローカル通知
  -> AudioProcessingService
    -> NativeAudioProcessor
      -> AudioFileService
      -> SpectralDSP
      -> 倍音/シマー解析
      -> foldover補完量推定
      -> ハム/低域ゴロゴロ整理
      -> ProcessingRoutePlanで補正ルートを決定
      -> ノイズ除去
      -> SpectralGateDenoiserでノイズ除去本体を実行
      -> サ行/シマー保護
      -> denoise後の再解析
      -> 最低限の高域修復
      -> 低中域の残りを軽く抑制
      -> シマー制限
      -> ピーク保護
    -> MasteringService
      -> LoudnessMeasurementService
        -> 公式K-weighting係数のLUFS / True Peak / 正式条件のLRA / 表示用の短期ラウドネスを測定
      -> MasteringAnalysisService
        -> 1回のFFT結果を帯域解析と刺さり感解析で共有
      -> ProcessingRoutePlanでマスタリングルートを決定
      -> MasteringProcessor
      -> MasteringSignalMathでマスタリング内の共通計算を担当
      -> MasteringHighFloorPreserverでマスタリング内の高域保持を担当
      -> de-esser
      -> 4バンドEQ
      -> 空気感
      -> 仕上げ
      -> 高域保持
      -> プリセット方針内の最終音量復帰
      -> 最終ノイズ確認
      -> 演奏中の150Hz〜500Hzを最終保護
      -> 最終音量上限確認
      -> 必要な時だけ短時間サ行ピーク制限
```

## コードベースの構造
- `Sources/VelouraLucent/App/`
  - アプリの起動です。Dock アイコンの上書きもここで行います。
- `Sources/VelouraLucent/Views/`
  - 画面です。差分サマリー、スペクトログラム、比較表示もここです。
  - `ContentView.swift` は、左の工程一覧、中央の作業画面、右の詳細設定をつなぎ、入力選択、補正開始、マスタリング開始、表示解析、書き出しの流れを担当します。
  - `VelouraSidebarView.swift` は、`音源` と `工程` の2セクションをつなぎます。
  - `VelouraMainWorkspaceView.swift` は、中央の作業画面と固定下部を担当します。横長波形、再生中スペクトル、ベクトルスコープ、横長スペクトログラム、直近ログ、全体工程をつなぎます。
  - `AudioWaveformWorkspaceView.swift` は、A/B再生、音量、ラウドネス合わせ、入力・補正後・最終版の横長波形を担当します。
  - `AverageSpectrumComparisonView.swift` は、再生中の入力・補正後・最終版から取ったスペクトルを表示します。表示名は再生中スペクトルです。
  - `VectorScopeView.swift` は、再生中の1音源について左右チャンネルの関係を `Polar Sample`、`Polar Level`、`Lissajous` で表示します。標準のセグメント切り替え、クリップ点、相関メーター、L/Rバランスメーター、履歴リセットもここで扱います。停止中は枠と説明だけを表示します。
  - `LoudnessMeterView.swift` は、ベクトルスコープ右側のラウドネスメーターを担当します。Momentary LUFS、Short-Term LUFS、Integrated LUFS、True Peak dBTP、目標線、上限線を表示します。安全、注意、超過のような総合判定は出さず、上限超過は該当メーターの値と線の色で確認します。
  - `SpectrogramComparisonView.swift` は、入力、補正後、最終版のスペクトログラムを横長3段で表示します。
  - `WorkspaceFooterView.swift` は、下部の直近3件と全体工程を担当します。補正とマスタリングの実進捗率もここへ渡します。
  - `RecentProcessingLogView.swift` は、日時、処理内容、対象ファイル、解析値、進捗を、直近3件まで表示します。
  - `OverallWorkflowView.swift` は、入力解析、補正処理、マスタリング、書き出しの現在位置と実進捗率を表示します。
  - `VelouraInspectorView.swift` は、開閉できる右設定パネルとして、右側の確認値、品質警告、完了後レポート、詳細設定を表示します。
  - `InspectorAnalysisPanel.swift` は、選択中音源のラウドネス、True Peak、ダイナミクス、ステレオ幅、品質警告を担当します。
  - `InspectorSettingsPanel.swift` は、右設定パネル専用の縦型設定画面を担当します。補正、マスタリング、アプリ設定を切り替え、解析モードはアプリ設定内に表示します。
  - `TermHelpButton.swift` は、設定項目や比較項目の `?` 説明ポップアップを担当します。
  - `SidebarFileRow.swift` は、3音源のファイル名、保存場所、形式、サンプルレート、チャンネル、長さを担当します。
  - `SidebarProcessingStatusView.swift` と `SidebarProcessStatusRow.swift` は、補正とマスタリングの現在位置、経過時間、進捗率、全工程一覧を担当します。
  - `ProcessingLogView.swift` と `FullProcessingLogView.swift` は、補正ログとマスタリングログの全文を同じ幅の2列で表示します。

## その技術を選んだ理由
- `SwiftUI`
  - Mac アプリの見た目を素直に組みやすいからです。
- `AVFoundation`
  - macOS 標準の音声読み書きが使えるからです。
- `UserNotifications`
  - macOS 標準の通知バナーと通知音を使えるからです。
- `Accelerate`
  - 周波数処理を Swift のまま実装しやすいからです。
- 段階式の自動処理
  - 非エンジニアでも、少ない選択で仕上がりを変えやすいからです。

## よくあるバグと修正方法
- 音声ファイルを開けない
  - 壊れたファイルか、対応外の形式の可能性があります。
- 出力ファイルができない
  - 元ファイルの場所に書き込み権限があるか確認します。
- MP3やM4Aにビット深度が表示されない
  - 圧縮音声にはWAVと同じ意味のビット深度がないためです。形式名、サンプルレート、チャンネル、長さを確認します。
- 左サイドバーで長いファイル名や保存場所が途中までしか見えない
  - 狭い幅で画面全体を押し広げないため、中央を省略しています。マウスを重ねると完全な内容を確認できます。
- 詳細ログの片側が見切れる
  - 補正ログとマスタリングログを同じ幅の2列にし、長い行を横へ伸ばさず折り返しているか確認します。
- `mp3` や `flac` 入力で書き出しに失敗する
  - 内部出力は `wav` に固定しています。利用者が保存するファイルは、選んだ用途に応じて `wav` または `m4a` になります。保存名の拡張子が選んだ用途と合っているか確認します。
- マスタリングが押せない
  - 先に補正を実行して、`*_lifter` を作ります。
- ノイズ除去を強くしすぎると音が細く聞こえる
  - まず `標準` で試して、必要な時だけ `強い` に上げます。
- 高域補完が強く聞こえる
  - 入力によっては倍音が少ないため、補完の効き方が変わります。
- 高域のザラつきが気になる
  - ノイズ除去の強さを上げると、高域の粒子感をより抑えやすいです。
- 高域が痛い
  - 詳細設定の `ハーシュネス抑制` を少し上げます。
- マスタリング後にシュワシュワ感が戻る
  - 通常は高域戻りガードを強くかけません。気になる場合は `エアー帯域` や `ハーシュネス抑制` を見直します。
- マスタリング後に音量が小さい
  - 最後に音量を再確認し、プリセットごとの音量変更上限、True Peak、ノイズ戻りを守れる範囲だけ戻します。ピーク余裕が足りない時は、無理に上げません。
- マスタリング後に煌びやかさや空気感が落ちる
  - 高域保持が、軽い 8kHz〜16kHz の帯域測定で確認します。ノイズが戻りすぎない範囲で最終版の高域を残します。
- 補正後に煌びやかさや空気感が落ちる
  - 補正後高域保持が、8kHz〜16kHz の実レベル低下を見て少し戻します。ヒス、シマー、サ行、16kHz以上の超高域が戻りすぎる候補は選びません。
- マスタリング後にサ行だけ強い
  - 補正後にサ行が強く残っている時は、最後に 5kHz〜9kHz の短時間ピークだけを軽く抑え、原音からの増えすぎを防ぎます。
  - 高域全体を下げるのではなく、短時間だけ目立つサ行ピークを対象にします。
- マスタリング後に高域が暗くなる
  - ノイズ戻りガードが、静かなヒスと短時間の高域チラつきを分けて確認します。8kHz〜20kHz が一度に落ちすぎる候補は選びません。
- 前のマスタリング結果が残って見える
  - マスタリング開始時に古い最終版プレビューを消し、新しい結果ができるまで書き出し対象にしません。
- 品質チェックで補正後の音量低下が出る
  - 補正後は音量を作る工程ではないため、まず情報として見ます。最終版でも低い時だけ注意として見ます。
- 品質チェックで高域低下やこもりが出る
  - `8kHz〜12kHz` の煌びやかさ、`12kHz〜16kHz` の空気感、`300Hz〜1kHz` のこもりを、入力、補正後、最終版の実測差分で見ています。
- ノイズチェックの表示が厳しく見える
  - `目立つ`、`少し目立つ`、`目立つ問題なし` で見ます。横棒の長さで、入力、補正後、最終版のどこで減ったか、どこで戻ったかを確認します。
  - 全体判定は、入力の悪さではなく、補正後や最終版に残っているノイズを基準にします。
  - `聴いて気になる場合の調整候補` は、実際に動かせる設定だけを出します。現在値とおすすめ値が同じになる時は表示しません。
  - 数値上の追加候補がない時も、最終版を聴いて違和感がないか確認する案内を出します。
  - ハムは低音全体ではなく、50/60Hzと倍音だけを周辺の周波数と比べます。低音楽器をそのまま電源ノイズ扱いしないためです。
- マスタリング後に広がるのに芯が弱い
  - `ステレオ幅` を下げます。いまは低域を広げない設計ですが、中高域は広がります。
- グラフが分かりにくい
  - まず上段で全体像を見て、次に再生中スペクトル、スペクトログラム、短期ラウドネス、ダイナミクス推移、相関メーターを確認すると分かりやすいです。
- 処理が重い
  - 長い音声や高負荷の環境では時間がかかります。同じ音声を共有できる場所では、読み込みとFFTを重ねないようにします。
  - ルートログの `実行工程数` と `スキップ工程数` を見ると、軽い素材でどの工程が減ったか分かります。
- 完了通知が出ない
  - アプリの「完了通知」設定、またはmacOSの通知許可が切れている可能性があります。必要に応じてONにします。
- どの工程で音が痩せたか分からない
  - 診断用の出力先を渡して、補正とマスタリングの中間WAVを工程ごとに書き出します。
  - 数値だけで判断せず、入力、補正途中、補正後、マスタリング途中、最終版を同じ音量感で聴き比べます。
- テストでノイズ値だけ良く見える
  - `Tests/Fixtures/AudioQuality/` の固定テスト音源を使い、ノイズ低下だけでなく、8kHz〜12kHz の煌びやかさ、12kHz〜16kHz の空気感、16kHz〜20kHz の超高域が残ることも確認します。

## 落とし穴と回避方法
- 圧縮音声へWAVのビット深度を当てはめない
  - MP3やAACは圧縮方式の名前を表示します。取得できない値を推測して画面へ出しません。
- 直近3件と完全な処理ログは役割が違います
  - 固定下部は今の状況を短く確認する場所です。全工程の記録は詳細ログに残します。
- 狭いサイドバーで工程名を長く説明しすぎない
  - 左サイドバーには全工程を出します。ただし各行は短い工程名にし、長い説明や完全なログは詳細ログで確認します。
- 処理は完全な AI モデルではありません
  - 高域補完は foldover を含む簡潔なヒューリスティック処理が中心で、補完量だけを軽い推定器で調整します。
- 補正は解析結果に応じて変わります
  - 倍音量、シマー、立ち上がり量、ノイズ量で効き方が少し変わります。
- マスタリングは別工程です
  - 補正と仕上げを分けて考えるほうが確認しやすいです。
- マスタリングの数値はアプリ内で同じ基準にそろえています
  - LUFS、True Peak、LRA は同じ測定サービスで出します。厳密な配信納品では外部メーターでの最終確認もあると安心です。
- Crest と LRA が下がりすぎると平坦になります
  - 比較表で `Crest` と `LRA` を見て、補正後から最終版で落ちすぎていないか確認します。
- 短期ラウドネスの線が横一直線に近いと、抑揚が減っている可能性があります
  - 最終版の線だけでなく、補正後との差を見ます。
- 再生中スペクトルは、いま再生している音源の線を見ます
  - 曲全体の平均ではなく、再生中の場所で周波数ごとの強さを見ます。
  - 入力、補正後、最終版をA/Bで切り替え、特に 8kHz から 12kHz あたりを聞き比べます。
- ノイズチェックは再生中スペクトルだけでは判断しません
  - 入力、補正後、最終版を、ヒス床、サ行瞬間差、シマー床、低域床、部屋床などの専用の測定値で見ます。
  - ノイズ床は、曲全体の平均音量ではなく、静かな区間を中心に見ます。音楽成分をノイズと間違えにくくするためです。
  - 横棒は変化を見やすくする補助です。正確な判断は、右側の実測値、差分、判定文で行います。
- サ行と高域のチラつきは別に見ます
  - `サ行・歯擦音` は発音の刺さり、`高域のチラつき` は高域の細かい揺れとして分けます。
- 改善アドバイスは、聴いて気になる場合に触る設定だけを出します
  - 補正で十分下がった時は補正設定を勧めません。マスタリングで戻った時は、マスタリング側の設定を優先します。
- ヒスと低域ゴロゴロは、静かな区間の床で見ます
  - ヒスは8kHz以上、低域ゴロゴロは20Hz〜150Hzの静かな区間を見て、補正の効き方を確認します。
- シマー制限のログで `軽量測定` が出た時は、長い音源の代表区間を使って測定しています
  - 強い補正では最後に全体確認も行い、速さだけで高域ノイズを残さないようにします。
- ノイズ戻りガードのログで `軽量判定` が出た時は、重い全体測定ではなく軽い判定で反復しています
  - 最後に全体確認を行い、ヒス、サ行、シマーの戻りをまとめて確認します。
- マスタリング後に `高域保持` のログが出た時は、ノイズ対策で高域が落ちすぎたため、音質バランスを保つ範囲で少し戻しています。
- `高域保持` は完全に取り消さず、ノイズが強い時でも最低限だけ残します。
- 補正後に `補正後高域保持/8-12kHz` や `補正後高域保持/12-16kHz` のログが出た時は、補正工程で痩せた煌びやかさや空気感を少し戻しています。
- `最終音量復帰` のログが出た時は、ノイズ戻りガード後に下がった音量、または安全に使える音量の余裕を、プリセットごとの目安超過幅を超えない範囲で、プリセットごとの音量変更上限、ピーク、ノイズを確認しながら最大2.0dBまで戻しています。目安超過幅は、自然と安全AI配信が0.75dB、聴きやすく整えるが1.0dB、YouTube / Spotify向けが1.25dB、押し出し強めが1.5dB、リリース音圧重視が2.0dBです。
- `ノイズ戻り: 緊急サ行上限` のログが出た時は、補正後から残ったサ行の短時間ピークだけを最後に抑えています。高域全体を暗くする目的ではありません。
- シマー制限の1回あたり最大削り量は、弱め2dB、標準3dB、強め4dBです
  - 短時間だけ飛び出るシマーを抑えつつ、持続する8kHz〜16kHzの煌びやかさと16kHz以上の空気感を残すことをテストで確認します。
- 相関メーターが0より左へ寄ると、モノラル再生で音が痩せる可能性があります
  - ステレオ幅を上げた時は、最終版の相関値も確認します。
  - 時間ごとの相関推移では、0より下に落ちた時間帯を確認します。長く0未満にある場合は、ステレオ処理や広がりを見直す候補です。
  - 無音区間は相関値として計算せず、前後の推移線をつなぎません。モノラル音源では左右の広がりを比較できないため、推移グラフではなく説明文を表示します。
- ダイナミクス保持と仕上げの強さは逆方向に効きます
  - 開放感を残したい時は `ダイナミクス保持` を上げ、前に出したい時だけ `仕上げの強さ` を上げます。
- 刺さり対策は高域全体の明るさと別です
  - `エアー帯域` を上げすぎたら、必要に応じて `ハーシュネス抑制` も調整します。
- 入力ごとに効き方が変わります
  - いつも同じ結果にはなりません。
- 出力形式は入力そのままではありません
  - `mp3` や `flac` を読んでも、内部のプレビュー用ファイルは `wav` になります。利用者が保存する時は、書き出しメニューで用途を選びます。
- 長尺ファイルは重くなります
  - まず短い音で効き方を確認すると安全です。
- 完了通知は処理完了時だけ送ります
  - 進捗更新のたびに通知すると作業の邪魔になるため、補正とマスタリングの最終完了だけを対象にします。
- ルート判定は処理開始直後に1回だけ決めます
  - 途中で判定が変わると、ログと実処理がずれやすいためです。
- 迷う素材は安全側で実行します
  - 速度よりも音質を優先し、必要な工程を飛ばさない設計です。

## ベストプラクティス
- サイドバーを変更した時は、未選択、処理中、完了の各状態を確認します。画面幅を変えても、左の音源と工程、中央の表示、右の設定が見切れないことを確認します。
- 固定下部を変更した時は、直近3件の長いファイル名、解析値、進捗バーが同時に収まるか確認します。
- 詳細ログを変更した時は、補正とマスタリングの両方に長い行を入れ、閉じる操作、折り返し、縦スクロールを確認します。
- まずは `./script/build_and_run.sh` で起動します。
- アイコンを変える時は `Resources/AppIcon-1024.png` を差し替えます。
- Xcode から直接起動した時の Dock アイコンも、起動時に同じ画像へ上書きします。
- `swift build` と `swift test` を通してから変更を重ねます。
- 音声処理は「補正後」と「最終版」を分けて聞いて確認します。
- 補正が遅い時は、まず工程別ベンチでどの工程が重いか確認します。
- 軽い素材で処理速度を見たい時は、ルートログの `実行工程数` と `スキップ工程数` を確認します。
- `ノイズ除去` は `標準` を基準にして、必要な時だけ変えます。
- マスタリング詳細設定は、まず `ハーシュネス抑制` と `中低域` から触ると分かりやすいです。
- 解析を増やす時は、まず既に読み込んだ `AudioSignal` を使えるか確認します。
- 周波数帯の集計を増やす時は、毎回 `filter` せず、先にビン範囲を計算して使い回します。
- 比較表示に必要な値だけなら、重い解析サービスを丸ごと呼ばず、既存のFFT結果や軽い計算を使います。
- 短期ラウドネスやLRAのような音量窓は、サンプル列を毎回合計し直さず、累積エネルギーから計算します。
- LUFS、True Peak、LRA を使う時は `LoudnessMeasurementService` を通します。画面、グラフ、品質チェックで値がずれないようにするためです。
- `SpectralDSP.stft` を触る時は、補正、解析、スペクトログラムの全部に影響するため、関連テストと実測を必ず確認します。
- `SpectralDSP.istft` を触る時も、補正結果そのものに影響するため、パイプラインテストと実測を必ず確認します。
- `Spectrogram` の中身を触る時は、フレーム番号と周波数番号の位置ズレが起きないように、A/B確認を必ず行います。
- スペクトログラムの色を変更する時は、入力、補正後、最終版で同じ `0〜-100表示dB` を使います。低域と高域で色の基準がずれないよう、周波数バケット幅に合わせて表示値を補正します。
- ベクトルスコープを変更する時は、音声処理中の受け取り処理から画面状態を直接変更しません。計算結果だけを別の処理列から渡し、画面更新はメイン処理で行います。これを守らないと、再生ボタンを押した時にアプリが終了する原因になります。
- ベクトルスコープの軌跡を消す条件を変更する時は、停止、自然終了、ファイル変更をまとめて確認します。一時停止では軌跡を保持するため、停止との動作を混同しないようにします。履歴リセットは、再生中の音源のベクトルスコープ履歴だけを消します。
- ノイズ除去のmask式を触る時は、式を変えず、前計算版と従来式が一致するテストを先に置きます。
- mask適用の最適化では、ノイズ除去の式と出力を変えず、参照の仕方だけを軽くします。
- 並列化する時は細かく分けすぎず、波形、チャンネル、FFTのような大きなまとまりにします。
- マスタリング解析の数値を増やす時は、`magnitudes()` で大きな配列を作る前に、既存のスペクトログラムを直接走査できるか確認します。
- 解析や表示で周波数の強さが必要な時は、全フレーム分をまとめて作らず、`fillMagnitudes` や `fillMagnitudeHistory` で必要な分だけ使います。
- `medianFilter` を触る時は、旧実装と完全一致するテストを先に通してから使います。
- チャンネルごとの補正を並列化する時は、結果の順序を保ち、左右が入れ替わらないようにします。
- 差分を見る時は、入力・補正後・最終版の3つを同じ場所で見比べると理解しやすいです。
  - 音源変化を見る時は、短期ラウドネスで音量感、再生中スペクトルとスペクトログラムで帯域の形、ダイナミクス推移で押し固まり、相関メーターと時間ごとの相関推移で位相の安全性を確認します。
- 完了後レポートは、右側の音声確認エリアにあるボタンから開きます
  - LUFS、True Peak、音量変化、ノイズ、高域保持を1か所にまとめます。
  - 新しい解析は増やさず、入力、補正後、最終版の既存解析結果だけを使います。
  - 数値だけで合格や不合格を決めず、最終判断は試聴で行います。
- ノイズチェックを増やす時は、帯域の平均音量ではなく、静かな区間のノイズ床、瞬間的な突出、50/60Hzの周辺比のように測定方法を先に決めます。
- ノイズチェックの横棒を変更する時は、グラフを消さず、値、改善量、戻り量、判定が同じ行で読める形を保ちます。
- 進行状況を変える時は、ログ文の文字列判定ではなく `ProcessingProgressEvent` を使います。ログは人が読む説明、イベントは画面を動かす合図として分けます。
- 完了通知を変える時は、`ProcessingJob` から直接OS機能を呼ばず、`CompletionNotificationReporting` を通します。テストでは記録用の差し替え先を使い、本物の通知を送らずに確認します。
- 同じ音源のノイズ測定値は使い回し、同じ解析を何度も呼ばないようにします。
- 補正のハム検出では、50/60Hzと倍音の強さを測る時に、サンプルごとに三角関数を呼ばず、前回の値から次の値を計算します。判定式と出力は変えず、同じ測定を軽くします。
- 補正後のマスタリング用解析がすでにある時は、マスタリング開始時に再解析せず使い回します。
- 解析を使い回した時は、ログの `解析: 既存結果を使用` で確認します。
- マスタリングのノイズ戻りガードは、ヒス、サ行、シマーを一括で見て、途中経過と測定回数をログに出します。
- マスタリングのノイズ戻りガードは、最大3回の軽量判定で補正を進め、必要な時だけ最後に全体確認を1回行います。
- マスタリングでは、補正後から5kHz以上が落ちすぎないよう、5-8kHz、8-12kHz、12-16kHz、16kHz以上を分けて高域保持します。
- マスタリングの高域保持では、変化しない補正後音源と元音源の4帯域測定値を最初に1回だけ計算し、高域保持と最終高域保持で使い回します。処理途中で変化する音は、その時点の音を測ります。
- 最終音量復帰後もヒス、サ行、高域チラつきが大きく戻った時だけ、緊急上限ガードで弱く抑え直します。補正後よりヒスや高域チラつきが2.5dBを超えて戻った場合は、補正後基準の上限を守る範囲まで抑え直します。補正後で高域が大きく失われ、原音参照で高域を戻す素材では、原音より同じ種類のノイズが増えない範囲で、音楽的な高域として原音参照を優先します。ヒスは復帰前も確認し、復帰前が上限内なら、高域を追加で削らず音量復帰前へ戻します。復帰前も上限を超える場合は、既存の緊急上限ガードを続けます。
- 内部判定監査表は `Docs/InternalAudioJudgementAudit.md` にまとめています。
- 補正の低域ノイズ整理とシマー制限も、測定回数をログで確認できます。
- 補正のノイズ除去ログは、解析済みのSTFT結果から高域メトリクスを使い回し、ログ用の再STFTを避けます。
- マスタリング解析の帯域集計は、Metal対応Macでは本処理でMetalを使い、STFT全体を保持せず分割して計算します。ログでCPU経路かMetal経路かを確認できます。
- 無理に複雑な仕組みにせず、今の処理順を保って改善します。
